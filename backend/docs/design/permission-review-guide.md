## 0. 会议速讲卡

现场只需要记住下面六句话：

1. **为什么做：** 现在新增权限仍依赖研发写迁移或改数据，效率低，也容易出现脏数据
2. **做了什么：** 管理员可以在界面维护目录、列表页、隐藏页、Tab 和按钮权限
3. **怎么生效：** 创建权限节点后授权给角色，前端控制显示，后端 API 做最终鉴权
4. **怎么保证安全：** 权限 key 不可变，系统角色和系统节点受保护，高风险写操作必须经过 2FA
5. **为什么可靠：** 权限版本实时刷新，多表写入使用事务，并发移动、防重放和最后超管都有测试
6. **需要老板确认：** 角色管理是否只给可信管理员；如果要下放，必须增加可授予权限上限

三分钟版本可以直接这样讲：

> 目前系统已经有账号、角色、权限三层模型，这次不重建权限系统，只补齐权限节点的界面化
> 维护。以后目录、页面、Tab 和按钮权限可以在后台新增、编辑、启停、移动和删除，不需要
> 直接改数据库。创建后的权限 key 是前后端共同契约，不能随意修改；前端负责菜单和按钮
> 显隐，后端 API 负责最终鉴权。权限节点写操作统一经过 2FA，并保护超级管理员、系统权限树
> 和最后一个可用超管。角色和账号的多表写入使用事务，权限修改可以通知在线用户自动刷新。
> 当前唯一需要确认的边界，是角色管理先只开放给可信管理员，还是本期同时增加普通管理员的
> 可授予权限上限。

## 1. 开场怎么说

建议用下面这段话开场：

> 这次权限改造的核心，不是重新做一套权限系统，而是在现有 RBAC 基础上补齐可视化维护能力。
> 以后新增目录、列表页、隐藏页、Tab 或按钮权限，可以直接在管理界面定义和授权，不需要
> 开发人员手工修改数据库。同时，前端负责展示控制，后端继续负责最终鉴权，高风险操作统一
> 经过 2FA，保证权限可配置但不能被随意破坏。

一句话结论：

> 在不改变现有“账号、角色、权限”模型的前提下，让权限从手工维护变成可视化、可审计、
> 可验证的标准流程。

## 3. 为什么要做

### 3.1 当前问题

原有权限结构可以完成角色授权，但新增权限节点仍偏向开发操作：

1. 新增菜单、Tab、按钮权限时，通常需要手工写迁移或直接操作数据库
2. 前端菜单、路由、Tab、按钮使用的权限 key 容易不一致
3. 操作人员无法在界面中查看和维护完整权限树
4. 删除、禁用或移动权限时，缺少统一的安全保护
5. 权限配置错误后，可能影响超级管理员和整个系统管理入口

### 3.2 业务影响

- 新业务接入权限的沟通和开发成本较高
- 权限调整依赖研发发布，响应速度慢
- 手工数据操作难以审计，也容易产生脏数据
- 只控制前端显隐时，存在绕过页面直接调用 API 的风险

### 3.3 本次目标

本次要建立一条标准流程：

```text
定义权限节点
  → 2FA 确认
  → 写入权限树
  → 授权给角色
  → 前端控制菜单/页面/Tab/按钮
  → 后端 API 独立鉴权
  → 自动刷新在线用户权限
  → 使用受限角色验收
```

---

## 4. 方案是什么

### 4.1 保留现有 RBAC

系统仍然使用三层关系：

```text
管理员账号 → 角色 → 权限节点
```

- 管理员通过角色获得权限
- 一个角色可以拥有多个权限
- 权限节点通过父子关系组成权限树
- 不新增复杂的权限引擎或组织模型

### 4.2 支持五类权限节点

| 类型 | 用途 | 示例 |
| --- | --- | --- |
| 目录 | 侧边栏分组，不直接进入页面 | 系统管理 |
| 列表页 | 可进入的业务主页面 | 账号管理 |
| 隐藏页 | 不显示在侧边栏的详情、新增、编辑页 | 编辑账号 |
| Tab | 页面内部的业务分类 | 待审核 |
| 按钮 | 页面或 Tab 内的具体操作 | 审核通过 |

它们统一存储在 `menus` 权限树中，用稳定的权限 key 与前后端代码对接。

### 4.3 权限 key 是稳定契约

示例：

```text
orderManage
orderManage-edit
orderManage-tabReview
orderManage-tabReview-approve
```

权限 key 创建后不可修改，原因是它同时被以下位置引用：

- 数据库权限节点
- 角色授权关系
- 前端路由
- 前端 Tab 和按钮
- 后端 API 鉴权
- 自动化测试

需要更名时，应新增新 key、完成代码和角色授权迁移，再停用旧 key，不能直接修改。

## 5. 权限新增的完整流程

这是会上最适合重点讲的一段。

### 第一步：确认业务权限点

产品和开发先确认页面中需要区分的权限：

```text
谁可以看列表？
谁可以进入详情或编辑页？
谁可以看到某个 Tab？
谁可以执行新增、删除、审核等操作？
对应 API 应该使用哪个权限？
```

### 第二步：在界面创建权限节点

拥有“菜单管理”权限的管理员打开权限节点管理界面：

1. 选择父节点
2. 选择目录、列表页、隐藏页、Tab 或按钮类型
3. 填写展示名称和唯一权限 key
4. 设置排序和启用状态
5. 提交前进行 2FA 验证

后端会校验父子类型、重复 key、循环父级和系统节点保护。

### 第三步：授权给角色

节点创建成功后：

1. 系统自动授权给 `superadmin`，避免新权限无人可维护
2. 管理员在角色编辑页勾选新权限
3. 后端自动补齐祖先节点

例如只选择：

```text
orderManage-tabReview-approve
```

系统会自动补齐：

```text
订单目录 → 订单列表 → 待审核 Tab → 审核通过按钮
```

### 第四步：业务代码接入同一个 key

权限配置不会自动生成业务代码。开发人员或 AI 需要在代码中使用同一个 key：

- 路由：`permissionKey`、`permissionParent`
- Tab：`TabsType.role`
- 按钮：`PermissionButton` 或权限 Hook
- API：`RequireAnyPermission`

### 第五步：权限实时生效

角色或权限变化后，后端递增 `permission_version`。

前端在后续响应中发现版本变化，会自动：

1. 重新获取侧边栏菜单
2. 清空页面权限缓存
3. 重新获取 Tab 和按钮权限

一般不要求用户退出后重新登录。

### 第六步：使用受限角色验收

至少验证以下组合：

- 有列表权限，无 Tab 权限
- 有某个 Tab，没有 Tab 内按钮
- 有按钮显示权限，API 同时允许调用
- 没有隐藏页权限，直接输入 URL 仍被拒绝
- 绕过前端直接调用无权 API，后端返回 `403`

## 6. 为什么方案可靠

### 6.1 前端显隐不是安全边界

前端控制菜单、Tab 和按钮是否显示，只是为了用户体验。

真正的权限判断在后端 API。即使用户绕过页面直接发请求，没有对应权限仍会返回 `403`。

### 6.2 系统角色和系统权限受保护

系统禁止：

- 重命名、禁用、删除 `superadmin`
- 覆盖 `superadmin` 的权限集合
- 非超级管理员给账号绑定 `superadmin`
- 禁用或降权最后一个启用的超级管理员
- 删除、禁用、移动系统管理基础权限节点

这样可以避免误操作后所有管理员都失去系统维护能力。

### 6.3 禁用父节点会让整棵子树失效

如果禁用“订单管理”列表页，其下面的 Tab、按钮和隐藏页会立即停止参与权限计算。

子节点原始状态不会被改写，重新启用父节点后可以恢复原配置。

### 6.4 高风险操作统一经过 2FA

权限节点新增、编辑、删除、启停和移动，以及管理员密码或 2FA 重置，都复用统一的六格
2FA 弹窗。

安全机制包括：

- challenge 绑定当前账号、动作和目标对象
- challenge 只能使用一次
- 同一个 TOTP 时间片不能重复执行高风险操作
- 10 分钟内连续失败 5 次会被限流
- 验证码和 challenge 不写入明文操作日志

### 6.5 多表写入使用事务

以下操作必须整体成功或整体失败：

- 创建权限节点、授权超级管理员、刷新权限版本
- 创建角色并保存权限
- 编辑角色并覆盖权限
- 创建或编辑账号并绑定角色

不会出现角色创建成功但权限没保存，或账号修改成功但角色没更新的半完成状态。

### 6.6 并发操作有保护

权限树移动时会在数据库事务内串行校验父链，避免两个管理员同时移动节点形成循环树。

最后一个超级管理员保护和 2FA 防重放也有并发集成测试，不只依赖前端按钮禁用。

## 7. 为什么没有做得更复杂

这次方案遵循“解决真实问题，不扩大系统边界”：

| 没有采用的方案 | 原因 |
| --- | --- |
| 重建权限引擎 | 现有 RBAC 足以承载菜单、页面、Tab 和按钮权限 |
| 增加组织、部门、租户模型 | 当前需求没有涉及数据范围和租户隔离 |
| 引入分布式锁 | 权限树写入低频，PostgreSQL 事务锁已覆盖竞争点 |
| 增加通用 Unit of Work 框架 | 当前只有明确的角色、账号复合写入场景 |
| 配置权限后自动生成页面和 API | 权限定义与业务实现职责不同，自动生成会扩大复杂度和风险 |
| 禁用父节点时批量改写所有子节点 | 会破坏子节点原配置，重新启用时难以恢复 |

对老板可以概括为：

> 该加固的地方做了加固，但没有为了权限功能去重构整个后台架构。

## 8. 方案边界

### 8.1 本次已经解决

- 任意权限节点的界面化维护
- 目录、列表页、隐藏页、Tab、按钮的分层授权
- 权限节点新增、编辑、删除、启停和移动
- 权限 key 不可变
- 前后端统一权限校验
- 在线权限缓存自动失效
- 系统角色、系统节点和最后一个超级管理员保护
- 2FA challenge、防重放和失败限流
- 事务、并发测试、权限验证中心

### 8.2 本次不解决

- 自动生成业务页面或 API
- 行级、字段级数据权限
- 部门、组织、租户隔离
- 权限申请审批流
- 临时授权和自动到期

### 8.3 当前剩余的重要风险

普通角色的权限集合尚未限制为“操作者自身权限的子集”。

目前只有 `superadmin` 分配被严格保护。如果未来把角色编辑能力下放给普通运营管理员，
需要先增加“可授予权限上限”，并决定角色授权、账号改绑角色是否也必须经过统一 2FA。

建议会上确认：

> 当前版本是否只向可信管理员开放角色管理？

如果答案是“是”，本方案可以按当前边界上线。

如果答案是“否”，可授予权限上限必须进入本期范围。

## 9. 演示流程

建议使用“权限验证中心”和测试角色演示，不直接删除系统权限。

### 9.1 会前准备

- 一个 `superadmin` 演示账号
- 一个受限测试账号
- 测试账号已绑定 2FA
- 提前准备以下权限组合之一：

```text
permissionLab
permissionLabOrders
permissionLabOrders-tabReview
permissionLabOrders-approve
```

### 9.2 演示顺序

1. 用超级管理员打开“角色与权限”
2. 展示完整权限树和五种节点类型
3. 打开权限节点管理，展示新增和编辑表单
4. 演示权限 key 在编辑状态不可修改
5. 在测试树中新增一个按钮权限，展示统一六格 2FA
6. 把按钮授权给受限角色
7. 切换到受限账号，展示只有已授权 Tab 和按钮
8. 取消按钮权限，展示在线权限刷新
9. 直接访问未授权隐藏页，展示无权限页面
10. 说明即使绕过前端，后端 API 仍返回 `403`

### 9.3 演示时不要做

- 不删除或禁用 `systemManage`
- 不修改 `superadmin`
- 不现场修改生产角色
- 不用真实业务账号验证
- 不把测试节点建在系统权限树中

## 10. 老板可能会问什么

### Q1：是不是以后任何权限都不用开发了？

不是。

权限节点和角色授权可以在界面配置，但业务页面、按钮和 API 仍需要开发，并使用同一个
权限 key 接入。我们解决的是权限定义和授权维护，不是业务代码自动生成。

### Q2：为什么权限 key 不能修改？

因为权限 key 同时被数据库、前端路由、按钮、Tab、后端 API 和测试引用。允许直接修改会
导致配置已经变化，但代码仍使用旧 key。需要更名时采用新增、迁移、验证、停用旧 key 的流程。

### Q3：前端把按钮隐藏就够了吗？

不够。

用户可以绕过页面直接调用接口，所以后端 API 必须再次校验权限。前端显隐只负责体验，
后端鉴权才负责安全。

### Q4：管理员误删系统权限怎么办？

系统角色和系统权限节点在前后端都有保护。绕过前端直接请求也会被后端拒绝，并且不能禁用
或降权最后一个启用的超级管理员。

### Q5：为什么每次权限操作都需要 2FA？

权限节点变更会影响一批用户的系统能力，删除和禁用还可能导致业务入口立即失效，因此把节点
写操作定义为高风险操作。统一 2FA 可以确认当前操作者身份，并防止验证码被重复利用。

### Q6：权限修改后需要重新登录吗？

通常不需要。

后端通过权限版本通知前端刷新菜单、路由、Tab 和按钮缓存。后端 API 权限本身也是实时查询，
不会因为前端缓存而继续放行旧权限。

### Q7：为什么不直接增加一个新的权限系统？

现有账号、角色、权限三层模型已经足够。本次真正缺少的是可视化维护、安全保护和可靠性，
重建权限引擎会增加迁移和维护成本，但不会带来对应业务价值。

### Q8：目前最大的剩余风险是什么？

如果把角色编辑权限下放给普通管理员，他们仍可能给普通角色授予自己没有的非系统权限。
因此当前建议只向可信管理员开放角色管理；进一步下放前增加可授予权限上限。

---

## 11. 会议结尾怎么说

建议这样收尾：

> 这套方案保留了现有 RBAC，不改变业务系统的基本结构，重点补齐了权限节点的界面化维护、
> 前后端统一校验、2FA、系统保护和事务一致性。对业务来说，新增权限不再依赖手工改数据库；
> 对安全来说，配置权限不等于绕过后端鉴权；对开发来说，权限 key、接口和验收流程都有统一
> 规范。当前需要确认的主要边界，是角色管理先只开放给可信管理员，还是本期同时建设普通
> 管理员的可授予权限上限。

会议最终需要形成三个结论：

1. 是否同意按现有 RBAC + 可视化权限节点管理方案上线
2. 是否同意产品内置权限由迁移固化、运行期权限由界面维护
3. 角色管理是否仅向可信管理员开放
