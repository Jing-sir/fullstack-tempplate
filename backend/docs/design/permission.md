# 权限系统架构设计

> 版本：1.0.0
> 更新：2026-05-31

---

## 一、整体思路

采用 **RBAC（基于角色的访问控制）** 模型：

```
管理员用户 → 绑定角色 → 角色拥有菜单/按钮权限 → 返回权限 key 列表 → 前端过滤路由和按钮
```

权限分两类：
- **菜单权限**（type=1）：对应前端路由的 `name`，控制侧边栏是否显示该菜单
- **按钮权限**（type=2）：格式为 `routeName-actionName`，控制页面内按钮是否可见

## 二、数据库表设计

### 2.1 admin_users — 管理员用户表

```sql
CREATE TABLE admin_users (
  id         BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  uid        VARCHAR(36)  NOT NULL UNIQUE,          -- 对外暴露的全局唯一 ID（UUID）
  username   VARCHAR(64)  NOT NULL UNIQUE,
  password   VARCHAR(255) NOT NULL,                 -- bcrypt 哈希
  email      VARCHAR(255) NOT NULL DEFAULT '',
  status     SMALLINT     NOT NULL DEFAULT 1,       -- 1=正常 0=禁用
  created_at TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ  NOT NULL DEFAULT NOW()
);
```

### 2.2 roles — 角色表

```sql
CREATE TABLE roles (
  id          BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  name        VARCHAR(64)  NOT NULL UNIQUE,   -- 角色标识，英文，如 super_admin / operator
  title       VARCHAR(64)  NOT NULL,          -- 显示名称，中文，如 超级管理员 / 运营
  description VARCHAR(255) NOT NULL DEFAULT '',
  status      SMALLINT     NOT NULL DEFAULT 1,
  created_at  TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
  updated_at  TIMESTAMPTZ  NOT NULL DEFAULT NOW()
);
```

### 2.3 menus — 菜单与按钮权限表

```sql
CREATE TABLE menus (
  id         BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  parent_id  BIGINT       NOT NULL DEFAULT 0,   -- 0 = 顶级，关联本表 id
  name       VARCHAR(128) NOT NULL UNIQUE,      -- 权限 key（见 2.6 命名规范）
  title      VARCHAR(64)  NOT NULL,             -- 显示名称
  type       SMALLINT     NOT NULL DEFAULT 1,   -- 1=菜单 2=按钮
  icon       VARCHAR(64)  NOT NULL DEFAULT '',  -- 仅菜单有效
  sort       INT          NOT NULL DEFAULT 0,   -- 同级排序，数字越小越靠前
  status     SMALLINT     NOT NULL DEFAULT 1,
  created_at TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ  NOT NULL DEFAULT NOW()
);
```

### 2.4 role_menus — 角色权限关联表

```sql
CREATE TABLE role_menus (
  role_id BIGINT NOT NULL REFERENCES roles(id) ON DELETE CASCADE,
  menu_id BIGINT NOT NULL REFERENCES menus(id) ON DELETE CASCADE,
  PRIMARY KEY (role_id, menu_id)
);
```

### 2.5 admin_user_roles — 用户角色关联表

支持一个管理员绑定多个角色（取并集）。

```sql
CREATE TABLE admin_user_roles (
  admin_user_id BIGINT NOT NULL REFERENCES admin_users(id) ON DELETE CASCADE,
  role_id       BIGINT NOT NULL REFERENCES roles(id) ON DELETE CASCADE,
  PRIMARY KEY (admin_user_id, role_id)
);
```

### 2.6 权限 key 命名规范

| 类型 | 格式 | 示例 |
|------|------|------|
| 菜单（父级） | `模块名` | `user`、`asset`、`systemManage` |
| 菜单（子级） | `页面路由name` | `userList`、`rolePermissions` |
| 按钮 | `路由name-操作名` | `userList-add`、`userList-export`、`rolePermissions-edit` |

规则：
- 全部使用**小驼峰**，与前端路由 `name` 保持一致
- 操作名固定词汇：`add`、`edit`、`delete`、`export`、`detail`、`audit`、`enable`、`disable`
- 禁止使用拼音、空格、特殊字符（`-` 仅用于菜单名和操作名之间的分隔）

## 三、API 接口设计

### 3.1 当前用户信息（登录后必调）

```
GET /api/v1/me
```

响应：
```json
{
  "data": {
    "uid": "xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx",
    "username": "admin",
    "email": "admin@example.com",
    "status": 1,
    "roles": ["super_admin"],
    "permissions": [
      "user", "userList", "userList-add", "userList-export",
      "systemManage", "rolePermissions", "rolePermissions-edit",
      "accountManage", "accountManage-add"
    ]
  }
}
```

`permissions` 是当前用户所有角色权限的**并集、扁平列表**，前端存入 store 后直接用于：
- 菜单过滤：`permissions.includes(route.name)`
- 按钮控制：`permissions.includes('userList-add')`

### 3.2 菜单管理

```
GET    /api/v1/menus             获取完整菜单树（管理页用，非当前用户权限）
POST   /api/v1/menus             新增菜单或按钮
PUT    /api/v1/menus/:id         更新
DELETE /api/v1/menus/:id         删除（级联删除子节点和 role_menus 关联）
```

### 3.3 角色管理

```
GET    /api/v1/roles             角色列表
POST   /api/v1/roles             新增角色
PUT    /api/v1/roles/:id         更新角色基本信息
DELETE /api/v1/roles/:id         删除（级联删除关联）
GET    /api/v1/roles/:id/menus   获取角色已授权的菜单 id 列表
PUT    /api/v1/roles/:id/menus   批量覆盖角色权限（传 menu_ids 数组）
```

### 3.4 管理员账号管理

```
GET    /api/v1/admin-users        列表
POST   /api/v1/admin-users        新增
PUT    /api/v1/admin-users/:id    更新（不含密码）
DELETE /api/v1/admin-users/:id    删除
PUT    /api/v1/admin-users/:id/roles  更新用户角色
```

## 四、前端权限消费方式

### 4.1 登录流程

```
登录成功 → 调用 GET /api/v1/me → 存入 permissionStore
→ 用 permissions 过滤 permissionRoutes → addRoute 注册 → 渲染侧边栏
```

### 4.2 菜单过滤规则

- 路由 `meta.ignorePermission = true`：直接放行（如首页）
- 路由 `meta.isShow = true`：隐藏路由，不进菜单，但需要 `requiresAuth`，不做权限过滤
- 其余路由：用路由 `name` 匹配 `permissions` 列表

### 4.3 按钮权限

使用自定义指令 `v-permission`：

```vue
<a-button v-permission="'userList-add'" @click="openAdd">新增</a-button>
```

或 composable：

```ts
const { hasPermission } = usePermission()
if (hasPermission('userList-export')) { ... }
```

### 4.4 `meta.role` 废弃说明

原 `meta.role` 字段废弃，全部改为直接用路由 `name` 作为权限 key。
迁移方式：删除所有路由 `meta` 中的 `role` 字段，后端菜单数据中的 `name` 对应路由 `name`。
