## 各大目录职责详解

### 1. `views/` — 页面层

**类比**：一栋楼里的各个房间

```
views/
├── Login/          登录页
├── Home/           首页
├── User/           用户管理
├── AssetManage/    资产管理
└── SystemManage/   系统管理
```

用户能在浏览器地址栏看到、点侧边菜单能进去的每一个页面都在这里。

每个页面按功能模块分文件夹，内部统一三层结构：

```
views/某个模块/某个功能/
├── Index.vue          页面主体
├── modal/             弹窗 / 抽屉
└── components/        页面内部的私有子组件
```

**好处**：找页面代码直接看文件夹名；找弹窗就进 `modal/`；1 个页面出了问题最多动 1 个文件夹，不会牵一发动全身。

---

### 2. `components/` — 公共组件库

**类比**：标准零件库

```
components/
├── TableSearchWrap/   搜索 + 表格 + 分页 三合一骨架（全项目最重要组件）
├── Header.vue         顶部导航栏
├── SideNavigationBar/ 左侧菜单
├── TagsView.vue       顶部页签（多标签页）
├── GoogleCode.vue     2FA 验证码弹窗
└── ...
```

被多个页面共用的 UI 零件都在这里。

**为什么要单独放**：如果项目有 30 个列表页，每个页面都自己写搜索框、表格、分页，那就是 30 份几乎一样的代码。一旦要改样式或逻辑，就要改 30 个地方。

**好处**：改一处，全项目生效。`TableSearchWrap` 尤其典型——所有列表页都用它，只需要给它传"搜索配置"和"列定义"就能生成完整页面，开发速度提升 3-5 倍。

---

### 3. `api/` — 接口层

**类比**：与服务器通话的专线

```
api/
├── api.ts           基础封装（axios 二次封装）
├── sys/             /sys/* 开头的接口（系统管理）
├── userApi/         /userAsset/* 开头的接口（用户资产）
├── flashExchange/   /flashExchange/* 开头的接口
└── ...
```

所有和后端服务器"打交道"的代码都集中在这里，按接口 URL 的前缀分组归档。

**为什么要单独放**：如果把 HTTP 请求代码直接写在页面里，以后后端改了接口地址或参数，你要全项目搜索在哪里调用过，可能改漏、改错。

**好处**：接口改了只动 `api/` 里的一个文件；页面代码只管"我要什么数据"，不关心"怎么拿"。职责清晰，排查问题快。

---

### 4. `store/` — 全局状态

**类比**：办公室的公告板

```
store/
├── user.ts      当前登录用户信息
├── sideBar.ts   侧边栏折叠状态 + 权限菜单树
├── tagsView.ts  顶部页签列表
└── theme.ts     主题色
```

需要被多个页面同时读写的"共享数据"存在这里。任何人都能看到上面的内容，任何人都能更新它，而不需要一个组件传话给另一个组件。

**好处**：登录后用户名和权限信息存一次，Header 组件、侧边菜单、权限按钮都能直接读，不用层层"父传子"地传递数据。

---

### 5. `use/` — 可复用逻辑

**类比**：最佳实践代码片段

```
use/
├── useConfirmAction.ts   确认操作弹窗（删除 / 禁用前的"你确定吗"）
├── useModalHandler.ts    弹窗开 / 关 / 提交生命周期管理
├── useFetchTableData.ts  表格数据加载逻辑
├── useFormHandler.ts     表单提交通用逻辑
├── useOnActivated.ts     页签切换时的刷新时机控制
└── ...
```

可以被多个页面"插入使用"的逻辑片段（Vue 3 里叫 Composable）。

**为什么要单独放**：每个页面都有"删除前弹确认框"的需求，如果每次都手写，逻辑一样但代码复制了 30 份。

**好处**：逻辑复用，风格统一。比如全项目的确认框样式和交互一致，产品体验更专业。

---

### 6. `utils/` — 纯工具函数

**类比**：函数工具箱

```
utils/
├── common.ts     通用工具（复制、防抖节流、深拷贝）
├── constant.ts   正则表达式、分页默认值等常量
├── table.ts      表格接口数据适配器
└── aesGcm.ts     加密 / 解密
```

和 Vue、业务无关的纯 JavaScript 函数。

**与 `use/` 的区别**：`utils/` 里的函数不依赖 Vue 响应式系统，任意地方均可调用；`use/` 里的 Composable 只能在 Vue 组件内使用。

**好处**：需要格式化金额、加密数据、防抖处理？直接 import，不用重复造轮子。

---

### 7. `routes/` — 路由配置

**类比**：页面地图 + 门禁系统

```
routes/
├── permissionRoutes.ts  权限路由（需要登录才能访问）
└── constantRoutes.ts    公开路由（登录页、错误页）
```

定义 URL 地址和页面组件的对应关系，同时控制"没权限的人不能进某些页面"。

就像一栋楼的平面图 + 门禁系统：平面图告诉你哪个门牌号对应哪个房间，门禁系统决定你的工牌能不能刷开某扇门。

**好处**：权限控制集中管理，不需要每个页面自己判断"我能不能被访问"。

---

### 8. `interface/` — 类型定义

**类比**：合同模板

```
interface/
├── TableType.ts          表格 / 搜索 / 列 / 操作按钮的类型定义
└── SideNavigationType.ts 侧边菜单节点类型
```

TypeScript 类型声明，描述数据的"形状"——有哪些字段、各自是什么类型。

就像合同模板：你不用每次都重新商量"这份数据里有什么字段"，模板已经定好了，双方按模板填就行。

**好处**：写代码时 IDE 会自动提示有哪些字段，写错字段名立刻报错，大大减少低级 Bug。

---

### 9. `enums/` — 枚举常量

**类比**：状态码翻译词典

```
enums/
├── statusText.ts    状态值 → 显示文案 + 颜色的映射
└── whitelistEnum.ts 白名单相关业务枚举
```

把后端返回的数字（如状态 `1`、`2`、`3`）翻译成人类可读的文案（"正常"、"冻结"、"注销"）和对应颜色。

**好处**：状态文案改动只改这一个文件，所有用到这个状态的页面自动更新。不会出现"这个页面改了，那个页面忘改"的问题。

---

### 10. `lang/` — 多语言

**类比**：翻译本

```
lang/
├── zh-CN.json   中文
└── en-US.json   英文
```

存所有界面上显示的文字，中英文各一份。

**好处**：切换语言不改代码，只换读取的 JSON 文件；新增语言只加一个新 JSON 文件即可。

---

### 11. `setup/` — 应用初始化

**类比**：开店前的准备工作

```
setup/
├── router-setup.ts   路由守卫、登录拦截、权限过滤
└── i18n-setup.ts     多语言系统初始化
```

应用启动时只需要跑一次的初始化逻辑。这些代码和具体业务无关，是"基础设施"。

**好处**：应用启动逻辑一目了然，新人上手能快速理解"项目是怎么跑起来的"。

## 架构设计的核心理念

### 原则一：单一职责

每个目录只做一件事。`api/` 只管请求，`store/` 只管状态，`views/` 只管页面。出了问题知道去哪找，不会满项目翻。

### 原则二：按"用途"分层，不按"类型"分

```
❌ 旧做法：all-components/  all-utils/  all-hooks/   （混在一起，越来越乱）
✅ 本项目：components（UI）  use（逻辑）  utils（工具）  （职责不重叠，新加功能知道该放哪）
```

### 原则三：公共能力下沉，业务逻辑上浮

`TableSearchWrap`、`useConfirmAction` 这些"底层能力"沉到公共层，业务页面只做"配置"，不做"实现"。

结果：30 个列表页的搜索 + 表格，统一由一个组件驱动，改一处全部生效。

## 一个新功能的完整开发路径

以"新增一个用户管理列表页"为例，代码流向如下：

```
1. 后端接口定义  →  src/api/userApi/xxx.ts       （接口层）
                          ↓
2. 数据类型声明  →  src/interface/TableType.ts    （类型层，按需扩展）
                          ↓
3. 状态枚举配置  →  src/enums/statusText.ts       （枚举层，按需扩展）
                          ↓
4. 路由注册      →  src/routes/permissionRoutes.ts（路由层）
                          ↓
5. 页面开发      →  src/views/User/xxx/Index.vue  （视图层）
                    ↙              ↘
              modal/ 弹窗      components/ 私有子组件
                          ↓
6. 复用公共能力  →  TableSearchWrap（搜索表格组件）
                    useConfirmAction（确认框 hook）
                    useModalHandler（弹窗管理 hook）
```

每一步都有明确的落点，不需要"猜"代码应该放哪里。
