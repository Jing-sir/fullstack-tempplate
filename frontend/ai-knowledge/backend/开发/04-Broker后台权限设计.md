# Broker后台权限设计

## 1. 文档目标

基于 `upay-broker-api` 的接口结构，对 Broker 后台进行权限模型设计，输出可落地的 RBAC（Role-Based Access Control）方案，覆盖：

- 菜单权限（页面可见）
- 按钮/操作权限（功能可用）
- 数据权限（可见范围）
- 审批权限（高风险动作复核）

> 本文以接口前缀与业务域为核心进行权限分组，便于前后端一致化实现。

---

## 2. 权限模型设计

### 2.1 核心实体

- **用户（Member）**：后台登录账户
- **角色（Role）**：权限集合载体
- **权限点（Permission）**：最小授权单元（建议到“接口动作级”）
- **数据范围（DataScope）**：约束用户可访问的数据边界

### 2.2 建议关系

- 用户与角色：多对多
- 角色与权限点：多对多
- 角色与数据范围：一对一或一对多（按业务域细分）

### 2.3 权限编码规范（建议）

统一格式：

`broker:{domain}:{resource}:{action}`

示例：

- `broker:sys:member:list`
- `broker:wallet:withdraw:review`
- `broker:customer:group:edit`
- `broker:card:operate:lock`

---

## 3. 权限域划分（基于 upay-broker-api）

以下按接口前缀归类为后台功能模块。

### 3.1 系统管理（sys）

对应接口：

- `/v1/sys/member/**`
- `/v1/sys/role/**`
- `/v1/sys/permission/**`

建议权限点：

- 成员管理：查询、新增、编辑、删除、启停、重置密码、重置谷歌验证
- 角色管理：查询、新增、编辑、删除、启停
- 权限管理：菜单权限查看、权限树查看

高风险操作（建议二次验证/审批）：

- 重置密码
- 重置谷歌验证
- 角色权限变更

### 3.2 团队管理（team）

对应接口：

- `/v1/team/**`

建议权限点：

- 团队列表、团队维护（新增/编辑）
- 成员加组、换组、设组长、解散/删除

数据权限建议：

- 仅本团队
- 本团队及下级
- 全部团队

### 3.3 客户管理（customer）

对应接口：

- `/v1/customer/**`
- `/v1/customer/group/**`

建议权限点：

- 客户查询、详情、编辑、冻结/解冻
- 客户分组管理（增删改查）
- 分组成员维护（加入/移除）
- KYC信息查看
- 导出能力（客户列表导出、分组导出）

高风险操作：

- 冻结/解冻
- 客户关键信息修改
- 大批量导出

### 3.4 卡与费率（card）

对应接口：

- `/v1/card/**`
- `/v1/card/rate/**`
- `/v1/card/deposit/**`

建议权限点：

- 卡操作：激活、锁卡、解锁、注销
- 卡消费权限设置/查询
- 卡交易记录查询、导出
- 费率组管理、费率维护、费率导出
- 卡充值记录查询、统计、导出

高风险操作：

- 锁卡/解锁/注销
- 费率修改

### 3.5 钱包与资产（wallet / asset / fiat）

对应接口：

- `/v1/wallet/deposit/**`
- `/v1/wallet/manager/**`
- `/v1/wallet/address/**`
- `/v1/asset/**`
- `/v1/asset/transfer/**`
- `/v1/fiat/asset/**`
- `/v1/bill/**`

建议权限点：

- 充值记录查询/导出
- 提现管理：审核、备注、重发、回退
- 地址簿查询/导出
- 资产查询、提现提交、提现校验
- 资产划转
- 法币账户提现/充值、账单查询/导出
- 账单明细查看、导出

高风险操作（建议强管控）：

- 提现审核（withdrawCheck）
- 重发转账（afreshTransfer）
- 回退金额（backTransferAmount）
- 提现提交（数字资产/法币）

### 3.6 收单/代付/汇率（acquirer / outlay）

对应接口：

- `/v1/acquirer/**`
- `/v1/acquirer/exchange/rate/**`
- `/v1/outlay/**`
- `/v1/outlay/exchange/rate/**`

建议权限点：

- 收单订单：查询、详情、导出
- 代付订单：查询、详情、导出
- 汇率列表查看

数据权限建议：

- 按商户/渠道/团队维度过滤

### 3.7 风控与合规（kyc / validate / callback / log）

对应接口：

- `/v1/kyc/**`
- `/v1/validate/**`
- `/v1/agent/callback/**`
- `/v1/agentOperateLog/**`

建议权限点：

- KYC查看与提交
- 验证能力（验证码校验等）
- 回调记录查看
- 操作日志查询、导出

高风险建议：

- 日志导出需单独权限
- 敏感字段脱敏显示

### 3.8 首页与公共能力（home / common / file）

对应接口：

- `/v1/home/data/**`
- `/v1/common/**`
- `/v1/file/**`

建议权限点：

- 首页看板查看
- 公共字典/地区信息查看
- 文件上传

## 4. 角色模板（建议初始化）

### 4.1 超级管理员（SUPER_ADMIN）

- 全模块读写
- 可管理角色与权限
- 可执行所有高风险动作

### 4.2 运营管理员（OP_ADMIN）

- 客户、团队、卡、资产、订单管理
- 不可改系统核心权限（角色/权限树）

### 4.3 财务管理员（FINANCE_ADMIN）

- 钱包、资产、账单、法币账户、提现流程
- 仅可读客户敏感信息（或按需脱敏）

### 4.4 风控审核员（RISK_AUDITOR）

- KYC、回调、提现审核、操作日志
- 不可进行资金发起类动作（仅审核）

### 4.5 客服/只读账号（CS_READONLY）

- 客户与订单只读
- 禁止导出、禁止状态变更

---

## 5. 数据权限设计

建议每个角色绑定数据范围策略：

- `ALL`：全部数据
- `TEAM`：本团队
- `TEAM_AND_CHILDREN`：本团队及下级团队
- `SELF`：仅本人数据
- `CUSTOM`：指定商户/渠道/团队集合

落地建议：

1. 登录后将 `memberId`、`roleIds`、`dataScope` 写入上下文（如 `AuthContext`）。
2. Service 查询层统一拼接数据过滤条件。
3. 导出接口复用同一过滤逻辑，避免“查不到但能导出”的绕过。

---

## 6. 高风险操作控制策略

以下操作建议启用至少 2 项控制：

- 二次身份验证（PIN/Google/短信）
- 双人复核（经办 + 审批）
- 冷却期（如修改后 N 分钟生效）
- 操作日志强制留痕（含前后值）

高风险清单建议覆盖：

- 提现审核与资金回退
- 费率变更
- 角色权限变更
- 密码/谷歌验证重置

---

## 7. 前后端联动建议

### 7.1 前端

- 登录后拉取菜单树（`/v1/sys/permission/homeMenu`）
- 路由守卫基于权限码控制页面访问
- 按钮级 `v-permission`（或同类指令）控制操作入口

### 7.2 后端

- 所有写操作接口必须做服务端权限校验
- 导出接口单独权限码，避免与查询权限混用
- 对敏感接口增加幂等、防重放、限频策略

---

## 8. 权限点样例（可作为初始化脚本输入）

| 模块 | 权限码 | 描述 |
|---|---|---|
| 系统成员 | `broker:sys:member:list` | 查询成员列表 |
| 系统成员 | `broker:sys:member:add` | 新增成员 |
| 系统成员 | `broker:sys:member:reset_password` | 重置密码 |
| 系统角色 | `broker:sys:role:update` | 编辑角色 |
| 客户 | `broker:customer:info:view` | 查看客户详情 |
| 客户 | `broker:customer:state:update` | 冻结/解冻客户 |
| 卡 | `broker:card:operate:lock` | 锁卡 |
| 卡费率 | `broker:card:rate:update` | 调整费率 |
| 钱包提现 | `broker:wallet:withdraw:review` | 提现审核 |
| 钱包提现 | `broker:wallet:withdraw:retry` | 重发转账 |
| 账单 | `broker:bill:export` | 账单导出 |
| 日志 | `broker:log:export` | 操作日志导出 |

---

## 9. 落地实施顺序（建议）

1. 建立权限码字典（菜单 + 按钮 + 接口）
2. 梳理角色模板并完成首批映射
3. 接口接入统一鉴权拦截器（含数据权限）
4. 接入操作日志与高风险动作二次校验
5. 联调前端菜单、路由、按钮权限
6. 做回归测试（重点：导出、审核、资金操作）

---

## 10. 结论

该设计以 `upay-broker-api` 的实际接口域为基础，形成“角色 + 权限点 + 数据范围 + 风险控制”的完整方案。建议优先上线“系统管理、客户管理、钱包提现、卡操作”四大核心域，随后逐步扩展到订单与汇率等辅助域，确保 Broker 后台权限治理可控、可审计、可扩展。
