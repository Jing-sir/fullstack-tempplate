# 异常码与异常处理

## 1. 文档目的与范围

本文基于 UPay 现有代码实现，梳理并规范以下内容：

- 异常码定义与分层
- 业务异常抛出方式
- 全局异常处理策略
- 多模块（App/Sys/Broker/Open/CaaS）差异与统一建议

涉及主要模块：

- `upay-common`
- `upay-app-api`
- `upay-sys-api`
- `upay-broker-api`
- `upay-open-api`
- `upay-caas-api`

---

## 2. 异常码体系现状

## 2.1 统一异常对象

UPay 统一使用 `AppServerException` 承载业务异常：

- 位置：`upay-common/src/main/java/com/upay/common/exception/AppServerException.java`
- 关键字段：
- `code`：业务码
- `message`：业务文案（后续可被国际化替换）
- `param`：占位参数（用于 `{0}`、`{1}` 等动态文案）

推荐抛出方式：

- `throw new AppServerException(ResultBrokerCodeEnum.AUTH_ERROR);`
- `throw new AppServerException(ResultCodeEnum.PWD_ERROR.getCode(), ResultCodeEnum.PWD_ERROR.getMsg(), remainTimes);`

## 2.2 异常码枚举分层

当前核心枚举：

- `ResultCodeEnum`：App 主业务域
- `ResultSysCodeEnum`：后台系统域
- `ResultBrokerCodeEnum`：Broker 域
- `ResultOpenApiCodeEnum`：OpenAPI 域
- `ResultCaaSCodeEnum`：CaaS 域
- `CommonCodeEnum`：少量公共场景码

## 2.3 状态码风格差异

- App/Sys/Broker/CaaS 默认成功码：`200`
- OpenAPI 对外成功码：`0`
- 实现方式：OpenAPI `EncryptControl` 会把 `JsonResult.code == 200` 转为 `0`

## 2.4 参数注解校验码（OpenAPI 特有）

OpenAPI 注解校验使用独立“参数校验码常量”：

- 位置：`upay-open-api/.../ParamValidateCodeConstants.java`
- 典型编码：
- `10300` 参数不能为空
- `10301` 参数值错误
- `10302` 参数格式错误
- `10303` 参数长度错误

其 `ExceptionHandler` 会把注解 message 当作编码解析，再做国际化文案映射。

## 3. 全局异常处理机制

## 3.1 统一处理入口

各 API 模块均使用 `@RestControllerAdvice` 全局异常处理：

- 优先处理 `MethodArgumentNotValidException`
- 业务异常处理 `AppServerException`
- 兜底处理 `HttpRequestMethodNotSupportedException`、`Exception`、`Throwable`

## 3.2 响应包装规则

统一返回 `JsonResult`，`error(...)` 分支应满足：

- `success = false`
- `code = 对应业务码`
- `msg = 业务文案（后续可国际化替换）`
- `data = null`（错误场景通常为空）

## 3.3 国际化与响应后处理

所有模块都通过 `ResponseBodyAdvice + EncryptControl` 做结果后处理（至少包含文案替换）：

- App/Sys/Broker/CaaS：按 `Language` + `i18n` 资源替换 `msg`
- OpenAPI：
- 除文案替换外，还支持响应体按商户配置进行 JWE 加密
- 设置 `X-UPA-REQUESTID`、`timestamp` 响应头

## 3.4 日志与响应联动

Broker/OpenAPI 会在请求属性中落响应对象，供拦截器落操作日志：

- `AGENT_OPERATION_RESPONSE`
- `AGENT_OPERATION_LOG`

异常响应同样会参与日志记录闭环。

## 4. 模块差异（现状）

## 4.1 Broker API

- 参数校验异常：聚合 `MethodArgumentNotValidException` 的 message 文本后返回
- 业务异常：直接透传 `AppServerException.code/message`
- 方法错误/系统错误：使用 `ResultSysCodeEnum.BAD_REQUEST/SERVER_ERROR`

## 4.2 App API / Sys API / CaaS API

- 参数校验异常：返回字段级错误 Map（序列化为 JSON 字符串）
- 业务异常：透传 `AppServerException`，支持占位参数格式化
- 文案统一通过 `EncryptControl` 国际化

## 4.3 Open API

- 参数校验处理最细：
- 支持 `MethodArgumentNotValidException`、`BindException`、`HttpMessageNotReadableException`
- 注解 message 使用编码（如 `10300`）进行 i18n 映射
- 成功码转换为 `0`
- 支持按商户公钥对响应 data 做 JWE 加密

## 5. 业务代码抛错规范

## 5.1 必须规则

1. 业务异常统一抛 `AppServerException`。
2. 优先使用本域 `Result*CodeEnum`，避免硬编码数字。
3. 需要动态文案时必须传 `param`，不要手工拼接字符串。
4. Controller 只做入参与上下文基础校验；强业务校验必须放 Service。
5. 不向客户端暴露堆栈、SQL、第三方原始错误内容。

## 5.2 推荐抛错顺序

1. 参数格式/必填（注解或显式校验）
2. 权限与登录态
3. 数据存在性与归属
4. 状态机校验
5. 外部依赖失败（映射成统一业务码）

## 5.3 示例（推荐）

```java
if (StringUtils.isBlank(req.getRoleId())) {
    throw new AppServerException(ResultBrokerCodeEnum.VALID_EXCEPTION);
}

if (!hasPermission) {
    throw new AppServerException(ResultBrokerCodeEnum.AUTH_ERROR);
}
```

```java
// 带占位参数
throw new AppServerException(
    ResultBrokerCodeEnum.PWD_ERROR.getCode(),
    ResultBrokerCodeEnum.PWD_ERROR.getMsg(),
    remainTimes
);
```

## 6. 异常码分配规范（建议）

## 6.1 基础段建议

- `200/0`：成功
- `4xx/5xx`：协议级或通用错误（保留）
- `10000+`：业务错误主段
- `9000~9999`：历史特殊段（保留兼容）

## 6.2 新增编码要求

1. 在对应 `Result*CodeEnum` 中新增，命名语义清晰。
2. 同步新增多语言 `i18n` 资源项。
3. 在接口文档标注错误码、触发条件、是否可重试。
4. 若对外接口（OpenAPI），同时确认 SDK/文档更新。

## 6.3 重复码控制（重要）

现有枚举中已出现“同一枚举内重复 code”的情况（示例）：

- `ResultCodeEnum`：`10177`、`10200`、`10203`
- `ResultSysCodeEnum`：`10130`、`10177`、`10200~10204`、`10409`、`10503`
- `ResultCaaSCodeEnum`：`10203`

规范要求：

1. 新增时禁止复用已有 code（同一枚举内）。
2. 建议增加 CI 校验脚本，构建期阻断重复码提交。

---

## 7. 全局异常处理规范（建议统一版）

## 7.1 处理优先级

1. `MethodArgumentNotValidException` / `BindException`
2. `AppServerException`
3. `HttpMessageNotReadableException`（JSON 格式错误）
4. `HttpRequestMethodNotSupportedException`
5. `Exception`
6. `Throwable`

## 7.2 响应约束

- 统一输出 `JsonResult`
- 统一做国际化映射
- 统一写入响应上下文（便于日志链路）
- 统一错误日志格式：`URI + code + message + trace`

## 7.3 外部异常映射

第三方异常统一映射为平台内部码，例如：

- `EXTERNAL_ERROR`
- `SERVER_BUSY_ERROR`
- `WALLET_EXTERNAL_ERROR`

禁止将第三方原始错误直接透出给客户端。

---

## 8. 研发落地检查清单

新增接口前后必须检查：

1. 是否定义了域内错误码（而非硬编码）。
2. DTO 是否补齐注解校验。
3. Service 是否有状态与归属校验。
4. 是否在异常处理中可正确国际化。
5. OpenAPI 是否完成参数码（103xx）与文档同步。
6. 是否触发重复码检测。
7. 是否有可观测日志（含 requestId/traceId）。

---

## 9. 后续优化建议

1. 建立“全仓唯一异常码台账”（至少保证同一域内唯一）。
2. 在 `Result*CodeEnum` 上自动化生成错误码文档（Markdown/JSON）。
3. 统一各模块参数校验错误返回结构，减少前端/接入方适配成本。
4. 为异常处理增加统一 traceId 出参（目前仅部分模块预留）。
5. 将“重复码检查 + i18n 缺失检查”纳入 CI 必检项。
