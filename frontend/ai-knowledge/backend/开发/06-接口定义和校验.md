# 接口定义和校验

## 1. 文档范围

本文基于 `UPayT` 代码仓中 Broker 相关模块当前实现整理，重点覆盖：

- 接口定义约定（URL、方法、Header、返回结构）
- 鉴权与权限校验链路
- 参数校验（注解校验 + 手工校验 + 业务校验）
- 系统管理域（成员、角色、权限）核心接口校验规则

主要参考模块：

- `upay-broker-api`
- `upay-broker-server/upay-broker-service`
- `upay-broker-server/upay-broker-client`
- `upay-common`

## 2. 接口定义约定

## 2.1 URL 与版本

- 统一使用 `/v1/**` 路径版本前缀（例如 `/v1/sys/member/login`、`/v1/sys/role/list`）。
- 业务按域拆分路径：
- 系统用户：`/v1/sys/member/**`
- 角色：`/v1/sys/role/**`
- 权限：`/v1/sys/permission/**`
- 钱包、卡、KYC、出入金等按各自资源路径维护。

## 2.2 HTTP 方法

- 查询接口以 `GET` 为主。
- 变更类接口以 `POST` 为主（新增、更新、删除、导出、提交审批等）。
- 存在少量“查询但使用 POST”（主要为了复杂筛选条件和分页对象）。

## 2.3 Header 约定

- `Token`：登录后必传（除 `@NoLogin` 接口）。
- `Language`：语言头，进入拦截器后通过 `LanguageEnum.getByCode` 标准化，默认 `en-US`。

## 2.4 统一返回结构

统一返回 `JsonResult<T>`，字段：

- `success`: Boolean
- `ts`: Long（当前实现为秒级时间戳）
- `data`: 业务数据
- `code`: 业务码
- `msg`: 文案

说明：

- Controller 正常返回 `new JsonResult<>(data)`。
- 异常由全局异常处理器统一转换为 `JsonResult.error(...)`。

## 2.5 返回文案国际化

- `EncryptResponseAdvice` 在响应出参阶段统一进入 `EncryptControl`。
- `EncryptControl` 按 `Language` 从 `i18n/{language}` 资源包重写 `msg`。
- `AppServerException` 支持参数化消息占位替换。

## 3. 鉴权与权限校验

## 3.1 请求入口链路

Broker API 核心链路：

1. `DecryptFilter`：读取并包装请求体，进行 XSS 清理，挂载原始请求参数到操作日志上下文。
2. `AuthorityInterceptor.preHandle`：登录态、账号态、权限校验。
3. Controller 执行业务。
4. `EncryptResponseAdvice`：统一处理返回文案。
5. `AuthorityInterceptor.afterCompletion`：写入操作日志（有 `@CollectOperationLog` 的接口）。

## 3.2 登录态校验

`AuthorityInterceptor` 中关键步骤：

1. 非 `@NoLogin` 接口必须有 `Token`。
2. 验证 Token 签名并解密出 `memberId`。
3. 从 Redis 登录缓存读取 `MemberLoginInfoRes`，并比对 token 一致性。
4. 校验代理商状态（禁用则清缓存并拒绝访问）。
5. 非初始化豁免接口，必须满足：
- 首次密码已设置（`initPasswordType=DISABLE`）
- Google 2FA 已绑定（`googleState=ENABLE`）
6. 非初始化豁免接口，还要求登录态已激活（`useState=ENABLE`，即已通过登录 Google 校验）。

## 3.3 URI 级权限校验

对非权限白名单接口，执行：

- `agentMemberSysClient.validateRoleAuth(authUri, roleIds)`

服务端逻辑：

1. 逐角色查询可访问 URI 列表（优先 Redis 缓存，其次 DB）。
2. 聚合为逗号拼接字符串。
3. 使用 `contains(uri)` 判断是否有权限，无则抛 `AUTH_ERROR(10003)`。

相关表：

- `agent_role`
- `agent_permission`
- `agent_role_permission`
- `agent_role_member`

## 3.4 无需登录与免权限

- `@NoLogin`：仅跳过登录校验（如登录接口、部分字典接口、测试 webhook）。
- `initExcludeUriList`：只跳过“首次密码/Google绑定前置校验”和后续权限校验。
- `authExcludeUriList`：登录后可访问，但不走角色 URI 权限判定，主要是基础下拉和公共配置接口。

## 4. 参数校验体系

## 4.1 JSR-303 注解校验

Controller 对大量入参使用 `@Valid/@Validated`，请求对象内使用：

- 基础注解：`@NotBlank`、`@NotNull`、`@Min`、`@Size`
- 自定义注解：
- `@ValidateEnum(cls=...)`：枚举值合法性，支持可选、忽略值、逗号分隔多值
- `@ValidateTimeStamp`：时间戳格式
- `@ValidateBigDecimal`：数值格式

异常由 `ExceptionHandler(MethodArgumentNotValidException)` 统一拦截并返回校验错误。

## 4.2 Controller 手工校验

系统管理域接口大量采用“显式 if 校验 + 抛业务异常码”方式，例如：

- `StringUtils.isBlank(...)` / `Objects.isNull(...)`
- 失败抛 `ResultBrokerCodeEnum.VALID_EXCEPTION`

典型场景：

- 登录参数（邮箱、密码）
- 角色新增/编辑（角色名、权限 ID 列表）
- 成员管理（id、state、roleId、teamId 等）

## 4.3 Service 业务校验

服务层补充强业务约束：

- 数据归属校验（成员/角色必须属于当前代理商）
- 默认角色/主账号保护（不可删除或禁用）
- 角色下存在成员不可删除
- 密码历史 6 个月复用限制
- Google、PinCode、验证码等安全校验
- 代理商状态、成员状态、账户状态校验

## 5. 系统管理域接口定义与校验

## 5.1 成员管理（`/v1/sys/member`）

关键接口与校验：

- `POST /login`
- 校验：邮箱、密码非空
- 业务：密码错误次数累计，超过阈值冻结
- `POST /validateGoogle`
- 校验：`googleCode` 非空
- 业务：2FA 验证失败次数累计，超过阈值冻结
- `POST /setLoginPassword`
- 校验：密码非空
- 业务：不可与当前密码相同，不可与 6 个月内历史密码相同
- `POST /updateLoginPassword`
- 校验：新旧密码非空，且不同
- 业务：旧密码正确 + 历史密码规则
- `POST /setPinCode`
- 业务：验证码校验、PIN 加密入库、错误计数清理、通知发送
- `POST /add`
- 校验：`email/name/roleId/teamId` 必填
- 业务：邮箱唯一、创建成员并绑定角色+团队
- `POST /update`
- 校验：`id/name/roleId/teamId` 必填
- 业务：重建角色与团队关系
- `POST /delete`
- 校验：`id` 必填
- 业务：主账号不可删，且级联删除团队/角色关系

## 5.2 角色管理（`/v1/sys/role`）

- `POST /list`
- 入参：`name/status/pageNo/pageSize`
- 逻辑：按代理商类别 + 系统角色(agent_id=0)+自定义角色联合查询
- `GET /permissions/list`
- 校验：`roleId` 非空
- 返回：当前角色已分配权限菜单
- `GET /choose`
- 返回：可选启用角色列表
- `POST /add`
- 校验：`roleName`、`permissionIds` 必填
- 逻辑：新增角色 + 批量写入 `agent_role_permission`
- `POST /update`
- 校验：`roleId`、`roleName`、`permissionIds` 必填
- 逻辑：更新角色基本信息，重建权限关系，删除角色权限缓存
- `POST /delete`
- 校验：`id` 非空
- 业务：默认角色不可删；角色下有成员不可删
- `POST /updateStatus`
- 校验：`id`、`status` 必填
- 业务：默认角色不可操作

## 5.3 权限管理（`/v1/sys/permission`）

- `GET /homeMenu`
- 入参：`language`（默认值在接口定义中为 `en_US`）
- 校验：当前登录上下文 `roleIds` 不可空
- 返回：首页菜单权限列表（按角色状态启用过滤）
- `GET /permissions/list`
- 入参：`roleId`（新增角色时可空）
- 逻辑：以 Owner 默认角色权限为上界，返回可配置权限树及勾选状态

## 6. 错误码与失败语义

高频业务码：

- `10001` 参数校验失败
- `10003` 权限不足
- `10005` 登录失效
- `10020` 账户异常
- `10021` 登录无效，请重新登录
- `50016` 不支持该操作
- `50045` 代理商主账号不能删除

异常来源：

- 参数注解失败：`MethodArgumentNotValidException`
- 业务失败：`AppServerException`
- 其他异常：统一转 `SERVER_ERROR`

## 7. 当前实现注意点（校验一致性）

## 7.1 语言码命名不完全一致

- 系统标准是 `en-US/zh-CN`。
- 个别接口默认值仍为 `en_US`，会被 `LanguageEnum` 回退为 `en-US`，建议统一写法。

## 7.2 权限匹配使用字符串包含

- `validateRoleAuth` 当前用 `authUriStr.contains(uri)` 判定权限。
- 建议改为“集合精确匹配”避免 URI 子串误判。

## 7.3 角色状态变更后的权限缓存一致性

- 角色权限缓存按角色 ID 存储 1 天。
- 当前仅在 `updateRole` 删除缓存，`updateStatus/deleteRole` 未显式清理。
- 建议在角色禁用/删除时同步删缓存，避免短期权限残留。

---

## 8. 建议落地规范（接口定义与校验）

建议统一执行以下规则：

1. 新增接口必须同时定义：入参 DTO、注解校验、业务码语义。
2. 能用注解表达的必填/枚举/格式，优先注解，减少 Controller 手工 if。
3. 业务归属校验必须放在 Service 层（不可只做 Controller 校验）。
4. 鉴权改为“URI 精确匹配 + 缓存主动失效”。
5. 导出接口与分页接口保持同一筛选参数模型，避免口径不一致。
