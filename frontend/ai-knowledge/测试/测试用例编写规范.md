# 测试用例编写规范

## 1. 用例结构标准
每个用例必须包含：

| 字段 | 必填 | 说明 |
|------|------|------|
| 用例ID | ✅ | TC-模块-序号 |
| 用例标题 | ✅ | [操作] + [对象] + [预期结果] |
| 前置条件 | ✅ | 执行前必须满足的状态 |
| 测试数据 | ✅ | 具体值或数据范围 |
| 测试步骤 | ✅ | 1. 2. 3. 格式，每步可验证 |
| 预期结果 | ✅ | 与步骤一一对应 |
| 优先级 | ✅ | P0(冒烟) / P1(核心) / P2(一般) / P3(低) |
| 用例类型 | ✅ | 功能/性能/安全/兼容性/边界/异常 |

## 2. 命名规范
- **格式**: `[模块]_[场景]_[预期结果]`
- **示例**: `登录_密码错误_提示用户名或密码错误`

## 3. 步骤编写原则
- 每个步骤**只能有一个操作**
- 使用**动词开头**：点击/输入/跳转/验证
- 避免模糊词：“正常操作”、“任意数据”

## 4. 断言规范
- 显式断言：明确预期值
- 隐式断言：描述状态变化
- 禁止：“系统正常”、“无报错”

## 5. 测试数据管理
- 硬编码数据：仅用于固定场景
- 变量化数据：使用 `${variable}` 标记
- 敏感数据：标注 `[需脱敏]`
