# 07 开发决策与维护

## 开发前先做的判断

每次接需求前，先回答这几个问题：

1. 当前是不是新打开的窗口？如果是，是否已经先阅读 `skills/upay-manage/SKILL.md` 和 `knowledge-base/README.md`？
2. 这次有没有新增独立路由？
3. 如果新增了，是挂已有父级路由，还是新建父级业务域？
4. 如果没新增路由，是不是只在原页面上加功能？
5. 原页面当前是 `config-overall` 还是手写页？
6. 后端权限是否已经准备好？
7. 如果没准备好，是否需要临时复用已有 `role`？
8. 新增代码是否补了必要注释？
9. 新增样式是否优先使用 `Tailwind CSS`？
10. 如果新增了列表接口，是否已经使用 `TableQueryParams<T>`、`TableDataPages<T>`，并按 `index.ts` + `index.type.ts` 拆分类型文件？

## 新窗口启动规则

这是项目新增的硬规则：

- 每次打开新的对话窗口，必须先阅读 `skills/upay-manage/SKILL.md`
- 每次打开新的对话窗口，必须先阅读 `knowledge-base/README.md`
- 读完入口文件后，再按当前任务继续补充阅读对应专题知识库
- 涉及路由 / 权限、`config-overall`、遗留页面、API / 请求层时，必须继续阅读对应专题后再开始写代码

## 页面分类决策

### new-surface

满足以下任一条件，都按新页面处理：

- 新增独立路由
- 新增 route-level 页面
- 新增独立菜单页
- 新增父级业务路由域

默认动作：

- 优先使用 `config-overall`
- 路由按项目约定接入

### legacy-modification

满足以下条件时，按老页面增量处理：

- 不新增独立路由
- 只是在现有页面内加功能
- 弹窗、抽屉、详情层只是当前页的一部分

默认动作：

- 严格继承原页面风格
- 不偷偷重构成第三种写法

## 新页面接入规则

### 方式一：新增子路由

- 修改现有业务域路由文件
- 页面继续放在当前业务目录下

### 方式二：新增父级业务域

- 新建父级路由模块
- 注册到 `src/routes/asyncRoutes.ts`

## 后端权限未完成时的临时策略

项目目前允许两种临时兜底：

- 新增子路由时，临时复用同模块已有 `meta.role`
- 新增父级业务域时，临时参考已有父级模块的父子权限结构

但这两种都只是临时方案，后端权限补齐后必须回改。

## 注释规则

这是项目硬规则：

- 所有新增代码都要加有意义的注释
- 注释重点写业务目的、权限兜底、字段转换、特殊分支原因
- 不写纯废话注释

尤其以下内容必须补注释：

- 临时 `role` 复用
- 复杂条件分支
- 特殊字段映射
- 与后端强耦合的约定

## 样式规则

这是项目新增的硬规则：

- 新增样式默认优先使用 `Tailwind CSS`
- 能在模板里通过 utility class 完成的样式，不再额外新增新的样式块
- 如果目标页面本来就依赖现有 `SCSS` 结构，可以做最小的样式补丁
- 但不要因为增量需求继续扩写大段新的 `SCSS`

## 新增 API 规则

后续新增 API 时，列表接口统一按下面约束执行：

- 列表请求参数统一使用 `TableQueryParams<T>`
- 列表返回数据统一使用 `Promise<TableDataPages<T>>`
- 公共分页类型统一从 `@/interface/StateType` 引入
- API 类型定义统一使用 `declare interface`、`declare type`
- 新增 API 模块统一采用目录化结构：`index.ts` + `index.type.ts`

补充说明：

- `src/api/wallet/collection/index.ts` 可以作为列表接口签名参考
- 历史已有 API 文件命名不统一是现状，不因为这条新规范批量回改老文件
- 如果只是给老 API 文件增量加方法，继续在原文件基础上修改，不强行重构目录结构

## 什么时候需要更新知识库

出现以下情况时，应该同步更新知识库：

- 新增了一个父级业务路由域
- 路由接入方式发生调整
- 权限策略发生变化
- `config-overall` 公共能力或标准结构变了
- 请求层、统一响应、加密规则有变化
- 新增了项目级通用开发约定

## 更新知识库的原则

- 优先基于真实代码文件来写
- 一份文档只讲一个主题
- 多写“为什么”和“怎么排查”，少写空泛定义
- 如果某个知识已经在 README 或 skill 里写得很清楚，就不要在这里机械重复
