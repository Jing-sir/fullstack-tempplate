# 04 遗留页面改动指南

## 什么时候按遗留页面处理

符合下面情况时，不要把需求当成新页面：

- 没有新增独立路由
- 只是往当前页面里加筛选、列、按钮、tab、弹窗、抽屉、详情层
- 二级弹窗页、详情弹层、抽屉页只是从当前页面打开，没有变成独立路由

这类需求必须继承原页面当前写法。

## 项目里常见的遗留页面形态

### 形态一：`defineComponent` + `a-form` + `a-table`

代表样本：

- `src/views/Banner/BannerList.vue`
- `src/views/Asset/AssetList.vue`

常见特征：

- `<script lang="ts">` + `defineComponent`
- 页面里自己维护 `formState`
- 页面里自己维护 `pagination`
- `NProgress` + 本地 `loading`
- 手动调用 API 拉列表
- 页面底部自己挂 `Pagination`
- 用 `buttonPermissions()` 控制按钮显隐

### 形态二：`script setup` 但仍是本地手写流

代表样本：

- `src/views/Wallet/System/Index.vue`

常见特征：

- 用 `<script setup lang="ts">`
- 仍然自己手写表单、表格、分页和请求流
- 通常会复用 `src/use/useTableConf.ts`

### 形态三：页面是旧壳，但局部弹窗已经改成 `script setup`

真实项目里这种情况很多，例如：

- 列表页仍是老式写法
- `Modal/AddOrEdit.vue` 已经是 `script setup`

这不代表整个页面可以顺手整体改造。

## 增量修改时应遵守的原则

- 先看当前文件本身怎么写
- 再看同目录最近的兄弟页面怎么写
- 最后才决定新增代码放在哪一层

优先目标是保持局部一致，而不是顺手现代化。

## 真实代码里的局部模式

### 表单重置

很多旧页会：

- 用 `formReset.value.resetFields()`
- 把分页重置到第一页
- 再重新查询

### 页面激活刷新

很多旧页在 `onActivated` 里会：

- 判断 `route.hash === '#no-refresh'`
- 否则触发重置或重新查询

如果原页已经有这套行为，新增功能要兼容，不要破坏。

### 弹窗打开方式

很多旧页会：

- 页面里维护 `isModalOpen`
- 页面里维护 `modalType`
- 页面里维护 `activeData`
- 通过 `handleOpenModal(record, type)` 打开

如果只是新增操作弹窗，优先沿用这一套。

## 不应该做的事

### 把 `defineComponent` 老页面整体改造成 `config-overall`

除非需求明确要求重构，否则不要这么做。

### 把 Options / defineComponent 页面强行改成 `script setup`

这类改动会扩大 diff，增加回归风险。

### 引入新的第三套表格或查询抽象

项目已经有 `config-overall` 和多种历史写法，再引入新风格只会让一致性更差。

## 如何判断可以少量清理

下面这些局部清理通常是可接受的：

- 为了新增字段补一个局部类型
- 为了复用当前页面逻辑，抽一个很小的本地方法
- 修掉与本次改动直接相关的明显错误

下面这些通常不建议顺手做：

- 全文件换成新 API 风格
- 大面积重命名变量
- 把分页、查询、弹窗流全部重构

## 实战建议

- 在旧页里新增功能时，先复制同页已有写法
- 权限按钮继续走 `buttonPermissions()`
- 查询 / 导出 / 弹窗 / 状态切换继续沿用原页面节奏
- 新增代码必须补业务注释，尤其是权限兜底、状态分支和字段转换

## 老页面样式增量规则

项目现在要求新增样式默认优先使用 `Tailwind CSS`，但在遗留页面里要和原结构一起判断：

- 如果只是给老页面补一个很小的视觉调整，且原页面已经依赖现有 `SCSS` 结构，可以在原样式上做最小改动
- 如果是新增一块新的 UI 或一组新的视觉样式，优先直接用 Tailwind utility class
- 不要因为一次小需求把整页老样式重写成 Tailwind
- 也不要在老页面里继续堆新的大段 `scoped scss`

## 特别注意

如果原页面本身已经是 `config-overall`，那它不属于这里的“老式手写页”，而是应继续按 `config-overall` 扩展。
