# 模拟需求：卡片管理新增“批量冻结卡片”入口

## 1. 需求背景

运营团队希望在“卡片管理”列表页中提升风险处置效率。当前对异常卡片只能逐条处理，操作成本较高，因此需要增加批量冻结能力。

目标是在现有卡片管理页面中，支持对选中的卡片进行批量冻结，并保留操作原因。

## 2. 需求类型判断

该需求**不新增独立路由**，而是在已有“卡片管理”页面内增强批量操作能力，因此应判定为：

- `legacy-modification`

落地原则：

- 沿用当前卡片管理页面现有实现风格；
- 不强行重构为新架构；
- 只做满足需求的最小增量修改。

## 3. 需求范围

页面范围：

- 菜单：卡片管理
- 页面：卡片资产管理
- 路由域：`src/routes/card/*`
- 视图目录：`src/views/CardManage/*`

## 4. 功能说明

### 4.1 列表增加批量选择

在卡片列表中支持勾选多条记录。

### 4.2 增加批量冻结按钮

在页面操作区新增“批量冻结”按钮。

按钮显示条件：

- 用户拥有对应按钮权限；
- 至少勾选一条可操作记录。

### 4.3 打开批量冻结弹窗

点击按钮后弹出确认弹窗，弹窗内容包括：

- 已选卡片数量；
- 冻结原因输入框；
- 确认 / 取消按钮。

### 4.4 提交冻结请求

点击确认后调用后端批量冻结接口。

提交参数示例：

```json
{
  "cardIds": [1001, 1002, 1003],
  "freezeReason": "命中异常交易排查"
}
```

### 4.5 成功后反馈

成功后需：

- 提示“批量冻结成功”；
- 关闭弹窗；
- 清空当前勾选；
- 刷新列表数据。

## 5. 权限要求

- 新增按钮权限 key：`batchFreezeCard`
- 若后端按钮权限暂未就绪，可临时复用卡片管理域已有相近权限，并在代码旁补充 TODO 注释，后续替换为正式权限。

## 6. 接口约定

建议在卡片相关 API 模块中新增批量冻结方法，不要直接在页面内发请求。

建议命名：

- `batchFreezeCard`
- 或与当前模块风格一致的同义命名

## 7. 交互细节

- 未勾选任何卡片时，按钮禁用；
- 冻结原因必填；
- 提交中按钮显示 loading，避免重复提交；
- 若选中记录中包含已冻结卡片，前端可按当前页面风格决定是否过滤或提示。

## 8. 验收标准

满足以下条件即可验收：

1. 卡片列表支持多选；
2. 页面出现批量冻结入口；
3. 可填写冻结原因并提交；
4. 调用接口成功后列表刷新；
5. 权限控制生效；
6. 改动保持与当前页面风格一致。

## 9. 开发备注

- 这是一个典型的已有页面增强需求；
- 不需要新增独立路由；
- 不建议引入新的全局 store；
- 不建议为单次需求抽象过度公共组件；
- 优先参考卡片域内已有批量操作、冻结、状态切换的相邻实现。
